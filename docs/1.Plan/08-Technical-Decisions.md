## ADR-001: 프레임워크 선택 (Next.js vs Vite + React)

**상태**: ✅ 승인됨
**날짜**: 2026-02-23
**결정자**: 이서연 (PL), 정하은 (Frontend)

### 맥락
프론트엔드 프레임워크 선택 필요. MVP는 클라이언트 전용이지만, 향후 확장성 고려 필요.

### 고려한 옵션

#### 옵션 1: Next.js 14 (App Router)
**장점**:
- SSR/SSG 지원 (SEO 유리)
- API Routes (간단한 백엔드)
- Vercel 배포 최적화
- Image Optimization 내장
- 풍부한 생태계

**단점**:
- 클라이언트 전용 앱에는 오버스펙
- 학습 곡선 (App Router 신규)
- 번들 크기 다소 큼

#### 옵션 2: Vite + React
**장점**:
- 빠른 개발 서버 (HMR)
- 단순한 설정
- 가벼운 번들
- 클라이언트 앱에 최적

**단점**:
- SSR/SEO 어려움
- 백엔드 추가 시 별도 설정
- Vercel 배포 최적화 부족

#### 옵션 3: Create React App
**장점**:
- 검증된 안정성
- 간단한 설정

**단점**:
- 개발 서버 느림
- 유지보수 중단 우려
- Webpack 기반 (복잡)

### 결정

**Next.js 14 선택**

**이유**:
1. **SEO**: 공유 시 OG Image 중요 → SSR 필요
2. **확장성**: Phase 2에서 URL 스크린샷 기능 → API Routes 활용
3. **생산성**: Vercel 배포 워크플로우 간소화
4. **학습**: 팀의 Next.js 경험 있음

**Trade-off 수용**:
- 초기 번들 크기는 Code Splitting으로 완화
- App Router 학습은 공식 문서로 해결

### 결과

**긍정적**:
- SEO 최적화 (공유 시 미리보기 이미지)
- Vercel 배포 10분 만에 완료
- API Routes로 향후 기능 추가 용이

**부정적**:
- 초기 학습 시간 소요 (2일)

### 참고
- [Next.js 14 공식 문서](https://nextjs.org/docs)
- [Vite 비교 분석](https://vitejs.dev/guide/comparisons.html)

## ADR-002: 상태 관리 라이브러리 (Zustand vs Redux)

**상태**: ✅ 승인됨
**날짜**: 2026-02-23
**결정자**: 정하은 (Frontend)

### 맥락
팔레트 상태, 히스토리 등 전역 상태 관리 필요.

### 고려한 옵션

#### 옵션 1: Zustand
- **장점**: 경량 (2KB), 간단한 API, TypeScript 지원, Middleware 풍부
- **단점**: Redux 대비 생태계 작음

#### 옵션 2: Redux Toolkit
- **장점**: 검증된 안정성, 강력한 DevTools, 대규모 앱에 적합
- **단점**: Boilerplate 많음, 학습 곡선, 번들 크기 큼

#### 옵션 3: Context API + useReducer
- **장점**: 내장 기능, 추가 의존성 없음
- **단점**: 성능 이슈 (리렌더), Middleware 없음

### 결정

**Zustand 선택**

**이유**:
1. **단순함**: MVP 규모에 적합
2. **성능**: Selector 기반 리렌더 최적화
3. **Persistence**: LocalStorage 연동 Middleware
4. **TypeScript**: 타입 안정성 우수

### 결과

**긍정적**:
- 코드량 50% 감소 (vs Redux)
- DevTools 지원으로 디버깅 용이
- LocalStorage 연동 10줄로 완료

**부정적**:
- Redux 대비 레퍼런스 적음 (해결: 공식 문서 충분)

### 참고
- [Zustand 공식 문서](https://github.com/pmndrs/zustand)

## ADR-003: CSS 프레임워크 (TailwindCSS vs CSS Modules)

**상태**: ✅ 승인됨
**날짜**: 2026-02-23
**결정자**: 최예린 (Designer), 정하은 (Frontend)

### 맥락
빠른 프로토타이핑과 일관된 디자인 시스템 필요.

### 고려한 옵션

#### 옵션 1: TailwindCSS
- **장점**: 유틸리티 우선, 빠른 개발, 디자인 토큰, Tree-shaking
- **단점**: 클래스명 길어짐, 학습 필요

#### 옵션 2: CSS Modules
- **장점**: Scoped CSS, 전통적 방식
- **단점**: Boilerplate, 일관성 유지 어려움

#### 옵션 3: Styled Components
- **장점**: CSS-in-JS, 동적 스타일
- **단점**: 런타임 오버헤드, SSR 복잡

### 결정

**TailwindCSS + shadcn/ui**

**이유**:
1. **속도**: 클래스만으로 스타일링
2. **일관성**: 디자인 토큰 기반
3. **shadcn/ui**: 재사용 컴포넌트 + Radix UI (접근성)
4. **팀 선호도**: 디자이너/개발자 모두 선호

### 결과

**긍정적**:
- 개발 속도 2배 향상
- 디자인 시스템 자동 적용
- 번들 크기 최소화 (사용한 클래스만)

**부정적**:
- HTML 클래스명 길어짐 (허용 가능)

### 참고
- [TailwindCSS](https://tailwindcss.com)
- [shadcn/ui](https://ui.shadcn.com)

## ADR-004: 색상 조작 라이브러리 (chroma.js vs Color.js)

**상태**: ✅ 승인됨
**날짜**: 2026-02-23
**결정자**: 정하은 (Frontend)

### 맥락
색상 변환, 조화 규칙 계산, 접근성 검증에 필요.

### 고려한 옵션

#### 옵션 1: chroma.js
- **장점**: 직관적 API, 작은 크기 (13KB), 대비 비율 내장
- **단점**: 마지막 업데이트 1년 전

#### 옵션 2: Color.js (Lea Verou)
- **장점**: 최신 색상 공간 지원, 정확도 높음
- **단점**: API 복잡, 크기 큼 (30KB)

#### 옵션 3: tinycolor2
- **장점**: 가벼움 (5KB)
- **단점**: 기능 제한적, 대비 비율 없음

### 결정

**chroma.js**

**이유**:
1. **기능 충분**: 조화 규칙, 변환, 대비 비율 모두 지원
2. **API 단순**: 학습 곡선 낮음
3. **크기**: 합리적 (13KB gzip)
4. **검증됨**: Coolors 등 유사 도구 사용

### 결과

**긍정적**:
- 조화 규칙 구현 1일 완료
- 대비 비율 계산 정확도 100%

**부정적**:
- 없음

### 참고
- [chroma.js 공식 문서](https://gka.github.io/chroma.js/)

## ADR-005: 이미지 색상 추출 (color-thief vs Vibrant.js)

**상태**: ✅ 승인됨
**날짜**: 2026-02-23
**결정자**: 정하은 (Frontend)

### 맥락
이미지에서 주요 색상 추출 필요.

### 고려한 옵션

#### 옵션 1: color-thief
- **장점**: 가볍고 빠름, 브라우저 호환성 우수
- **단점**: 알고리즘 단순 (MMCQ)

#### 옵션 2: Vibrant.js
- **장점**: 더 정교한 색상 추출, 역할별 분류 (Vibrant, Muted 등)
- **단점**: 크기 큼 (20KB), 느림

#### 옵션 3: 직접 구현 (Canvas API)
- **장점**: 의존성 없음
- **단점**: 개발 시간 소요, 버그 가능성

### 결정

**color-thief**

**이유**:
1. **속도**: 1초 내 처리 (5MB 이미지)
2. **크기**: 6KB (작음)
3. **충분함**: MVP에 필요한 기능 모두 제공

### 결과

**긍정적**:
- 추출 속도 빠름
- 사용자 피드백 긍정적

**부정적**:
- Phase 2에서 Vibrant.js로 전환 고려 (더 정교한 추출)

### 참고
- [color-thief GitHub](https://github.com/lokesh/color-thief)

**상태**: ✅ 승인됨
**날짜**: 2026-02-23
**결정자**: 이서연 (PL)

### 맥락
단위 테스트 및 컴포넌트 테스트 필요.

### 고려한 옵션

#### 옵션 1: Vitest
- **장점**: Vite 기반 (빠름), Jest 호환 API, ESM 네이티브
- **단점**: 생태계 작음 (신규)

- **장점**: 검증된 안정성, 풍부한 생태계

**Vitest**

**이유**:
1. **속도**: 테스트 실행 3배 빠름
2. **설정**: Next.js와 통합 간단
3. **호환성**: Jest 마이그레이션 쉬움

### 결과

**긍정적**:
- TDD 워크플로우 쾌적
- Watch mode 빠름

**부정적**:
- 없음

## ADR-007: URL 스크린샷 기능 연기

**상태**: ✅ 승인됨
**날짜**: 2026-02-23
**결정자**: 전체 팀

### 맥락
URL 입력 시 스크린샷 캡처 후 색상 추출 기능.

### 고려한 옵션

#### 옵션 1: MVP에 포함 (Puppeteer)
- **장점**: 차별화 기능
- **단점**: 백엔드 필요, 배포 복잡도 증가, CORS 이슈

#### 옵션 2: MVP 제외, Phase 2에 추가
- **장점**: 리스크 감소, 집중 가능
- **단점**: 초기 출시 차별점 감소

### 결정

**Phase 2로 연기**

**이유**:
1. **기술 리스크**: 백엔드 서버 필요 (Vercel Functions 제약)
2. **시간 부족**: 4주 내 안정적 구현 어려움
3. **대안 존재**: 이미지 업로드로 유사 경험 제공

### 결과

**긍정적**:
- MVP 집중 가능
- 안정적 출시

**부정적**:
- 초기 차별점 약화 (이미지 업로드로 보완)

### Phase 2 계획
- 백엔드: Next.js API Routes + Puppeteer
- 배포: Vercel Pro 플랜 고려 (Function 실행 시간 제한 해제)

## ADR-008: 다크모드 지원 연기

**상태**: ✅ 승인됨
**날짜**: 2026-02-23
**결정자**: 최예린 (Designer), 정하은 (Frontend)

### 맥락
다크모드 지원 여부.

### 고려한 옵션

#### 옵션 1: MVP에 포함
- **장점**: 사용자 편의성
- **단점**: 디자인 작업 2배, 테스트 복잡도 증가

#### 옵션 2: MVP 제외, Phase 3에 추가
- **장점**: 개발 집중

**Phase 3로 연기**

**이유**:
1. **우선순위**: 핵심 기능 먼저
2. **시간**: 디자인 작업 시간 부족
3. **사용자**: 색상 도구 특성상 라이트모드 선호 (색상 정확히 보기)

### 결과

**긍정적**:
- MVP 집중 가능

**부정적**:
- 일부 사용자 불편 (수용 가능)

## ADR-009: Package Manager (pnpm vs npm)

**상태**: ✅ 승인됨
**날짜**: 2026-02-23
**결정자**: 정하은 (Frontend)

### 맥락
패키지 관리 도구 선택.

### 고려한 옵션

#### 옵션 1: npm
- **장점**: 기본 제공, 안정적
- **단점**: 느림, 디스크 공간 많이 사용

#### 옵션 2: yarn
- **장점**: 빠름, Workspace 지원
- **단점**: v1/v2 혼란

#### 옵션 3: pnpm
- **장점**: 가장 빠름, 디스크 절약, Monorepo 지원
- **단점**: 일부 패키지 호환성 이슈

### 결정

**pnpm**

**이유**:
1. **속도**: 설치 3배 빠름
2. **공간**: 디스크 70% 절약
3. **향후**: Monorepo 전환 대비

### 결과

**긍정적**:
- 설치 시간 단축
- CI/CD 빌드 시간 감소

**부정적**:
- 없음 (호환성 문제 없었음)

## ADR-010: Monorepo 구조 제외

**상태**: ✅ 승인됨
**날짜**: 2026-02-23
**결정자**: 이서연 (PL)

### 맥락
프로젝트 구조 (Monorepo vs Single Repo).

### 고려한 옵션

#### 옵션 1: Monorepo (Turborepo)
- **장점**: 코드 공유, 통합 빌드
- **단점**: 설정 복잡, MVP에 오버엔지니어링

#### 옵션 2: Single Repo
- **장점**: 단순함, 빠른 시작
- **단점**: 향후 백엔드 분리 시 재구성 필요

### 결정

**Single Repo (MVP), Monorepo (Phase 2 고려)**

**이유**:
- MVP는 프론트엔드만 → Monorepo 불필요
- Phase 2 백엔드 추가 시 재검토
